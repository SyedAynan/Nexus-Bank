## 1. INTRODUCTION

NEXA is an enterprise-grade digital banking platform designed to provide a secure, intelligent, and production-ready solution for modern financial operations. The platform combines a high-performance Python backend built with FastAPI and a feature-rich React single-page application (SPA) frontend, delivering a complete banking experience from account management to AI-powered fraud detection.

The significance of this project lies in its unique approach to solving real-world banking challenges using fundamental Data Structures and Algorithms (DSA). Rather than treating DSA as academic exercises, NEXA integrates nine core data structures â€” Hash Tables, Binary Search Trees, Linked Lists, Stacks, Queues, Priority Queues, Graphs, Tries, and multiple Sorting Algorithms â€” directly into the banking engine. Each data structure solves a specific banking problem: Hash Tables enable O(1) account lookups, Graphs detect circular money transfers indicative of money laundering, and Priority Queues process loan applications by creditworthiness.

The project is relevant in today's fintech landscape where digital-first banking is the norm. Traditional banking systems rely on legacy monolithic architectures that are difficult to scale and maintain. NEXA addresses this by adopting a modern microservice-oriented architecture with containerized deployment, real-time WebSocket streaming, and comprehensive security hardening including JWT authentication with MFA, role-based access control, and tamper-proof audit logging.

The platform serves multiple stakeholders: end users who perform banking operations like deposits, transfers, and loan applications; compliance officers who monitor Anti-Money Laundering (AML) alerts and fraud patterns; administrators who manage users, system settings, and audit trails; and analytics viewers who access 13 types of financial charts and dashboards.

## 3. LITERATURE REVIEW / EXISTING SYSTEM

### Current Technologies in Digital Banking

The digital banking landscape is dominated by several categories of solutions. Traditional core banking systems like Temenos T24, Finacle by Infosys, and Oracle FLEXCUBE provide comprehensive banking functionality but are built on legacy architectures (often COBOL or Java EE) that are expensive to maintain and difficult to modernize. These systems typically cost millions of dollars in licensing and implementation fees.

Modern neobank platforms like Chime, Revolut, and N26 have disrupted the industry with mobile-first interfaces and API-driven architectures. However, their technology stacks are proprietary and not available for educational study or independent deployment. Open-source banking projects like Apache Fineract provide basic banking operations but lack modern UI frameworks, real-time capabilities, and AI-powered intelligence features.

### Limitations of Existing Systems

**Opacity of algorithms:** Commercial banking systems hide their data structure and algorithm implementations behind proprietary code. Students and developers cannot study how fundamental CS concepts like graph traversal or priority queues are applied in real banking scenarios.

**Monolithic architecture:** Many existing banking platforms use monolithic designs that make independent scaling of components impossible. If the fraud detection engine experiences high load, the entire application must be scaled rather than just the affected service.

**Limited real-time capabilities:** Traditional banking systems rely on batch processing for transaction analysis, often running overnight jobs for fraud detection. This creates a significant delay between suspicious activity and detection.

**Poor UI/UX:** Many open-source banking projects provide functional but visually basic interfaces that don't reflect modern design standards. The user experience is often an afterthought rather than a primary design consideration.

**Insufficient security layers:** Some educational banking projects implement basic authentication but lack production-grade security features like MFA, rate limiting, security headers (HSTS, CSP), and comprehensive audit trails.

### Key Implementation Details

**Authentication Flow:** The login process sends credentials via OAuth2PasswordRequestForm. The backend validates against bcrypt-hashed passwords, then returns a 202 Accepted response with an MFA challenge. The frontend presents an OTP input, and upon verification, the backend issues JWT access and refresh tokens. Token refresh is handled transparently by Axios interceptors on 401 responses.

**Transaction Processing:** When a user initiates a transfer, the BankingService performs the following steps: (1) retrieve both accounts from the HashTable in O(1), (2) validate balances, (3) execute the debit/credit, (4) record the transaction in both accounts' LinkedLists via O(1) prepend, (5) push undo operations onto both accounts' Stacks, (6) add the transfer edge to the ComplianceGraph, (7) run real-time fraud scoring, and (8) broadcast the update via WebSocket.

**Fraud Detection Pipeline:** The FraudEngine scores every transaction using six signals with configurable weights. The velocity signal uses a deque-based sliding window counting transactions in the last hour. The graph signal computes risk from the ComplianceGraph's out-degree and transfer volumes. Scores above configurable thresholds generate FraudAlert records with severity levels.

**Audit Hash Chain:** Each audit log entry computes `SHA-256(previous_hash + action + details + timestamp)`, creating a blockchain-inspired chain. The `verify_audit_chain()` method traverses the chain and recomputes hashes to detect tampering. This provides O(1) append and O(n) verification.

## 11. LIMITATIONS

**In-memory DSA state:** The core DSA structures (Hash Table, BST, Graph, etc.) operate in-memory and are not persisted across server restarts. While PostgreSQL provides persistence for ORM models, the in-memory DSA state is re-seeded from the database on each startup, which may cause temporary inconsistencies.

**Single-server architecture:** The current deployment runs as a single FastAPI instance. While the architecture is designed for horizontal scaling, true distributed operation would require shared state management (e.g., Redis-backed DSA or distributed message queues).

**Mock external services:** OAuth (Google, Apple, Microsoft), WebAuthn/Passkey, email delivery, and market data integrations are implemented as service stubs with static fallback data. Production deployment requires real API keys and provider configurations.

**Static FX rates:** The multi-currency service uses static exchange rates rather than live market feeds. Real-time FX would require integration with providers like ExchangeRate-API or Fixer.io.

**BST balance:** The AccountBST implementation is a standard (non-self-balancing) BST, meaning worst-case operations degrade to O(n) if accounts are inserted in sorted order. An AVL or Red-Black Tree would guarantee O(log n) in all cases.

## 12. FUTURE SCOPE

**Self-balancing BST:** Upgrade the AccountBST to an AVL Tree or Red-Black Tree to guarantee O(log n) operations regardless of insertion order, eliminating the current worst-case O(n) degradation.

**Distributed architecture:** Implement message queues (RabbitMQ or Apache Kafka) for inter-service communication, enabling horizontal scaling of individual engines like the FraudEngine during peak loads.

**Machine learning integration:** Replace the rule-based fraud detection signals with trained ML models (e.g., Isolation Forest, LSTM anomaly detectors) for more sophisticated pattern recognition using scikit-learn and TensorFlow.

**Mobile applications:** Develop native iOS and Android applications using React Native, sharing the existing API layer and extending it with push notification support via Firebase Cloud Messaging.

**Blockchain audit trail:** Enhance the current hash chain audit logging with actual blockchain technology (e.g., Hyperledger Fabric) for cryptographically verifiable, distributed audit trails.

**Live external integrations:** Connect real OAuth providers, Plaid for account aggregation, SendGrid for email delivery, and market data APIs for live investment tracking.
